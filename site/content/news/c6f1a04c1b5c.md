把一个智能体在本地跑起来，今天几乎已经不算难题。几段提示词、几次模型调用，再接上几个工具，一个看起来能办事的 Agent 很快就能成形。但这篇文章真正想提醒的是：真正昂贵的部分，从来不在这个 Demo，而在它被团队正式接入生产之后。

作者借用了谷歌当年讨论机器学习系统“隐性技术债务”的经典框架，指出智能体工程正在重演类似问题。企业一旦让 Agent 接入代码仓库、工单系统、监控平台、云资源和内部知识库，挑战马上从“能不能做”转成“谁来管、怎么控、出了问题怎么追”。模型本身只是中间很小的一块，外面包着的是一整圈基础设施。

文章把这些额外成本拆成几类。第一类是集成：不同团队各自接 GitLab、数据仓库、Kubernetes 和密钥系统，看似灵活，实际会带来权限不一致、维护重复和故障排查困难。第二类是上下文：Agent 依赖的服务归属、配置、历史决策会持续变化，静态文档很快过时，导致它在错误前提下继续执行。第三类则是注册表、评估和治理——当组织里的智能体数量比员工还多时，企业需要知道有哪些 Agent 在运行、它们拥有什么权限、是否在重复造轮子。

更棘手的是，智能体不是传统软件里完全确定的流程。它会推理、会反思，也会在相同任务上给出不同路径，这使得“评估”本身比测试普通服务复杂得多。你不只要判断答案对不对，还要判断它为什么这么做、是否可复现、在权限边界内会不会走偏。没有人机回路和可观测性，团队往往要等事故发生后才知道系统已经失控。

这篇文章的价值在于，它把当下很多团队对 Agent 的兴奋感往后推了一步：从“先接进来再说”，转向“先把底座补齐再扩张”。如果企业今年真准备把智能体从个人工具推进到部门级生产系统，这几笔技术债几乎都躲不过。
